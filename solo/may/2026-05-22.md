## 기술적 결정
- **페이지네이션 전략 — `Page<T>` 유지** ([#79](https://github.com/hkjbrian/gongu/issues/79))
  - 전체 repository가 이미 `Page<T>`로 구현되어 있고, 현 규모에서 COUNT 오버헤드는 병목 아님
  - `Page<T>`는 `hasNext()`를 포함하므로 무한 스크롤 UX에도 동작
  - 향후 병목 실측 시: 고객향 주문 목록부터 커서 기반 페이지네이션 전환 검토
- **Product 상태 전환 방식** — Spring `@Scheduled` 1분 주기 스케줄러 채택
  - TaskScheduler 동적 등록(B안), Redis Keyspace Notification TTL(C안) 대비 단순성·운영 부담 우선
  - 초 단위 정확한 오픈이 필요한 경우(선착순 등)는 클라이언트 측 조건 검사 또는 별도 메커니즘으로 대응
- **만료 UPCOMING 처리 방식** — 활성화 스케줄러에 `endAt > now` 필터 추가 (A안)
  - 만료 UPCOMING → CLOSED 전환은 close 스케줄러(#92)로 책임 분리
  - 단일 스케줄러가 activate + close 양쪽을 처리하면 단일 책임 원칙 위반 + 메서드명 불일치
- **JVM 타임존 고정 방식** — `application.yml` 커스텀 프로퍼티 + `AppConfig @PostConstruct`
  - `LocalDateTime.now(ZoneId.of(...))` 매 호출 하드코딩, 상수 선언 대신 JVM 레벨 고정으로 코드베이스 전반 적용
  - 배포 환경 타임존 설정에 무관하게 동작 보장
