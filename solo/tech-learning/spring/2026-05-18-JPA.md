# 기술 학습 기록

## 주제
> JPA 영속성 컨텍스트, flush, readOnly 트랜잭션, JDBC/HikariCP 동작 원리

## 핵심 질문
- JPA는 왜 사용하는가?
- Persistence Context는 어떤 역할을 하는가?
- flush는 언제 발생하고 commit과 무엇이 다른가?
- `@Transactional(readOnly = true)`는 어떤 효과가 있는가?
- Repository에 트랜잭션이 있는데 서비스 계층에 `@Transactional`을 거는 이유는 무엇인가?
- JDBC, PreparedStatement, HikariCP는 각각 무엇인가?
- JDBC batching은 언제 적용되는가?
- bulk query, native query, lock은 어떻게 이해해야 하는가?

## 이미 알고 있는 내용

**JPA의 필요성**
- 객체는 참조를 통해 연관관계를 맺고, 관계형 DB는 외래 키를 통해 연관관계를 맺는다.
- 이 둘 사이에는 패러다임 불일치가 존재한다.
- ORM은 객체와 관계형 DB 사이의 불일치를 줄여주는 기술이다.
- JPA는 자바 진영의 ORM 표준 명세이고, Hibernate는 대표적인 JPA 구현체이다.

**트랜잭션**
- `JpaRepository` 기본 구현체인 `SimpleJpaRepository`에는 기본 트랜잭션 설정이 존재한다.
- 기본 조회 메서드는 `readOnly = true`, 저장/삭제 등 변경 메서드는 일반 `@Transactional`로 동작한다.
- 서비스 계층에 `@Transactional`을 걸면 여러 Repository 호출을 하나의 트랜잭션으로 묶을 수 있다.

## 학습을 통해 정리한 내용

**JPA와 Persistence Context**

JPA는 객체 상태 변화를 영속성 컨텍스트에 관리하게 하고, 특정 시점에 SQL로 변환하여 DB와 동기화한다.

```text
Java Object
→ JPA
→ Hibernate
→ JDBC
→ Database
```

영속성 컨텍스트의 주요 기능은 다음과 같다.

1. 1차 캐시
2. 동일성 보장
3. Dirty Checking
4. 쓰기 지연
5. 지연 로딩 지원
6. flush 시점의 SQL 생성

같은 영속성 컨텍스트 안에서는 같은 엔티티를 다시 조회할 때 같은 인스턴스를 반환할 수 있다.

```java
Member m1 = em.find(Member.class, 1L);
Member m2 = em.find(Member.class, 1L);

m1 == m2; // true
```

**Dirty Checking과 flush**

Dirty Checking은 영속 상태 엔티티의 변경을 감지하는 기능이다.

```java
@Transactional
public void changeName(Long id) {
    Member member = memberRepository.findById(id).orElseThrow();
    member.changeName("newName");
}
```

명시적으로 `save()`를 호출하지 않아도, 영속 상태 엔티티가 변경되면 flush 시점에 update SQL이 실행될 수 있다.

flush는 영속성 컨텍스트의 변경 내용을 DB에 SQL로 동기화하는 작업이다.

```text
flush = SQL을 DB에 보냄
commit = 트랜잭션을 확정함
rollback = 트랜잭션 변경을 취소함
```

flush가 발생하는 대표 시점은 다음과 같다.

1. 트랜잭션 commit 직전
2. JPQL 실행 전
3. native query 실행 전
4. `em.flush()` 직접 호출 시

flush는 SQL 실행 비용, dirty checking 비용, DB 락/제약 조건 검사 비용을 유발할 수 있다.
대량 처리에서는 일정 단위로 `flush()`와 `clear()`를 함께 사용하는 방식이 자주 쓰인다.

```java
if (i % 1000 == 0) {
    entityManager.flush();
    entityManager.clear();
}
```

**rollback과 1차 캐시**

트랜잭션이 rollback되면 DB 변경은 취소된다.

하지만 이미 변경된 자바 객체의 필드 값이 자동으로 원래대로 돌아가지는 않는다.

```java
member.changeName("rollback-name");
transaction.rollback();

member.getName(); // 메모리 객체는 변경된 값을 가질 수 있음
```

rollback 이후 엔티티는 보통 detached 상태가 되므로, rollback된 트랜잭션에서 나온 엔티티를 재사용하는 것은 주의해야 한다.

**서비스 계층의 @Transactional**

Repository 메서드에도 트랜잭션이 있지만, 서비스 계층에 `@Transactional`을 거는 이유는 여러 DB 작업을 하나의 작업 단위로 묶기 위해서이다.

장점은 다음과 같다.

1. 여러 DB 작업을 원자적으로 처리한다.
2. 같은 Persistence Context를 사용한다.
3. 1차 캐시와 동일성 보장이 유지된다.
4. LAZY 로딩을 서비스 로직 안에서 안정적으로 사용할 수 있다.
5. flush, commit, rollback 시점을 서비스 단위로 제어할 수 있다.
6. 같은 트랜잭션 범위에서 DB Connection을 재사용할 수 있다.

핵심은 커넥션 재사용보다 **원자성과 영속성 컨텍스트 범위 보장**이다.

**@Transactional(readOnly = true)**

`readOnly = true`는 조회 전용 트랜잭션임을 표현하는 설정이다.

`SimpleJpaRepository` 기준으로 클래스 레벨에 `@Transactional(readOnly = true)`가 붙어 있기 때문에, 기본 조회 메서드는 readOnly로 동작한다.
반면 `save`, `delete`, `flush` 같은 변경 메서드에는 일반 `@Transactional`이 붙어 있다.

효과는 다음과 같다.

1. 조회 의도를 명확히 표현한다.
2. 서비스 메서드 전체에서 하나의 영속성 컨텍스트를 유지한다.
3. LAZY 로딩을 안정적으로 사용할 수 있다.
4. Hibernate flush mode 조정으로 불필요한 flush를 줄일 수 있다.
5. dirty checking 비용을 줄일 수 있다.
6. JDBC Connection 또는 DB에 read-only 힌트가 전달될 수 있다.

단, `readOnly = true`는 쓰기 방지 장치가 아니라 최적화 힌트에 가깝다.

readOnly 트랜잭션에서 엔티티를 수정하면 환경에 따라 다음처럼 동작할 수 있다.

```text
1. 오류 없이 종료되지만 update SQL이 나가지 않음
2. 명시적 flush 또는 modifying query에서 예외 발생
3. DB/JDBC driver가 쓰기를 거부하여 예외 발생
```

실무에서는 조회 메서드는 `readOnly = true`, 수정 메서드는 일반 `@Transactional`로 분리하는 것이 좋다.

**JDBC, PreparedStatement, HikariCP**

JDBC는 자바 애플리케이션이 DB와 통신하기 위한 표준 API이다.

```text
Spring Data JPA
→ JPA
→ Hibernate
→ JDBC
→ JDBC Driver
→ Database
```

JDBC도 표준 API이고, 실제 구현체는 각 DB 벤더가 제공하는 JDBC Driver이다.

```text
MySQL → MySQL Connector/J
PostgreSQL → PostgreSQL JDBC Driver
Oracle → Oracle JDBC Driver
H2 → H2 JDBC Driver
```

`PreparedStatement`는 SQL 틀을 먼저 준비하고 값을 파라미터로 바인딩하여 실행하는 JDBC 객체이다.

```java
String sql = "select * from member where email = ?";
PreparedStatement ps = connection.prepareStatement(sql);
ps.setString(1, email);
```

장점은 다음과 같다.

1. SQL Injection 방지
2. SQL 파싱 및 실행 계획 재사용 가능성
3. 타입 기반 파라미터 바인딩
4. JDBC batching에 활용 가능

HikariCP는 JDBC Connection Pool 구현체이다.
DB Connection을 매번 새로 만들지 않고 미리 만들어둔 커넥션을 빌려주고 반환받는다.

```text
@Transactional 메서드 진입
→ TransactionManager가 트랜잭션 시작
→ DB 작업 시 HikariCP에서 Connection 획득
→ 같은 트랜잭션 동안 같은 Connection 사용
→ commit 또는 rollback
→ Connection 반환
```

**JDBC batching**

JDBC batching은 여러 SQL을 하나씩 보내지 않고 묶어서 DB에 전달하는 최적화이다.

Hibernate batching 설정 예시는 다음과 같다.

```properties
spring.jpa.properties.hibernate.jdbc.batch_size=50
spring.jpa.properties.hibernate.order_inserts=true
spring.jpa.properties.hibernate.order_updates=true
```

batch가 잘 적용되려면 조건이 필요하다.

```text
같은 형태의 SQL이 여러 개 있어야 함
flush가 너무 자주 일어나면 batch 효율이 떨어짐
IDENTITY id 생성 전략은 insert batching에 불리함
DB driver가 batch 실행을 지원해야 함
엔티티 종류가 뒤섞이면 batch 효율이 낮아질 수 있음
```

**@GeneratedValue**

`@GeneratedValue`는 엔티티 식별자 생성 전략을 지정한다.

```text
IDENTITY
SEQUENCE
TABLE
AUTO
UUID
```

`IDENTITY`는 DB가 insert 시점에 id를 생성한다.
따라서 insert를 해야 id를 알 수 있어 쓰기 지연과 insert batching에 불리할 수 있다.

`SEQUENCE`는 DB sequence에서 id를 먼저 가져온 뒤 insert를 나중에 할 수 있다.
`allocationSize` 설정을 통해 id를 일정 범위만큼 미리 확보할 수 있어 성능 최적화에 유리할 수 있다.

**lock**

JPA 락은 크게 낙관적 락과 비관적 락으로 나뉜다.

낙관적 락은 `@Version`을 사용한다.

```java
@Version
private Long version;
```

DB row를 미리 잠그지 않고, 수정 시점에 version 값을 비교하여 충돌을 감지한다.
일반적으로 flush 또는 commit 시점에 version 충돌을 검사한다.

비관적 락은 DB 락을 직접 사용한다.

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("select m from Member m where m.id = :id")
Optional<Member> findByIdForUpdate(Long id);
```

조회 시점에 DB에 쿼리가 실행되며, 보통 `select ... for update` 형태로 row lock을 획득한다.

**bulk query와 native query**

bulk query는 한 번의 쿼리로 여러 row를 update/delete하는 작업 방식이다.

```java
@Modifying
@Query("update Member m set m.status = 'INACTIVE'")
int deactivateAll();
```

native query는 JPQL이 아니라 DB SQL을 직접 작성하는 방식이다.

```java
@Query(value = "select * from member where email = :email", nativeQuery = true)
Member findByEmailNative(String email);
```

둘은 비교 기준이 다르다.

```text
bulk query = 대량 수정/삭제 방식
native query = SQL을 직접 쓰는 방식
```

native query도 bulk query가 될 수 있다.

```java
@Modifying
@Query(
    value = "update member set status = 'INACTIVE'",
    nativeQuery = true
)
int deactivateAllNative();
```

bulk update/delete는 영속성 컨텍스트를 우회하므로, 실행 후 `clearAutomatically = true` 또는 `entityManager.clear()`를 고려해야 한다.

**equals/hashCode와 동일성 보장**

Persistence Context의 동일성 보장은 `equals()`나 `hashCode()`로 이루어지는 것이 아니다.

같은 영속성 컨텍스트 안에서는 엔티티 타입과 식별자를 기준으로 같은 인스턴스를 반환한다.

`equals/hashCode`는 다음 상황에서 중요하다.

```text
Set, Map에 엔티티를 넣을 때
detached 엔티티끼리 비교할 때
다른 트랜잭션에서 조회한 엔티티를 비교할 때
프록시 객체와 비교할 때
```

id 기반으로 구현하면 id 생성 전후에 hashCode가 바뀔 수 있으므로 주의해야 한다.

## 프로젝트 적용

- 조회 서비스에는 `@Transactional(readOnly = true)`를 사용한다.
- 수정 서비스에는 일반 `@Transactional`을 사용한다.
- 서비스 계층에서 트랜잭션 범위를 잡아 여러 Repository 호출을 하나의 Persistence Context로 묶는다.
- readOnly는 쓰기 방지 장치가 아니므로 조회/수정 메서드를 명확히 분리한다.
- 대량 처리에서는 flush 시점과 영속성 컨텍스트 크기를 관리한다.
- 연관관계는 기본적으로 LAZY를 우선 고려한다.
- N+1 문제는 fetch join, EntityGraph, batch size, DTO projection 등으로 해결한다.
- bulk update/delete 이후에는 영속성 컨텍스트와 DB 상태 불일치를 주의한다.
- 실제 SQL 로그를 확인하며 JPA 동작을 검증한다.

## 오늘의 결론

- JPA는 ORM 표준 명세이고, Hibernate는 대표적인 JPA 구현체이다.
- JDBC도 표준 API이고, 실제 구현체는 각 DB의 JDBC Driver이다.
- JPA의 핵심은 영속성 컨텍스트이며, 영속성 컨텍스트는 1차 캐시, 동일성 보장, dirty checking, 쓰기 지연, flush를 제공한다.
- flush는 DB에 SQL을 보내는 동기화 작업이고, commit은 트랜잭션 결과를 확정하는 작업이다.
- 서비스 계층의 `@Transactional`은 여러 Repository 호출을 하나의 트랜잭션과 Persistence Context 범위로 묶기 위해 필요하다.
- `readOnly = true`는 조회 의도 표현과 flush/dirty checking 최적화를 위한 힌트이다.
- HikariCP는 DB Connection을 효율적으로 재사용하기 위한 Connection Pool이다.
- JPA를 잘 사용하려면 객체 상태 변화와 실제 SQL 실행 시점을 함께 예측할 수 있어야 한다.
