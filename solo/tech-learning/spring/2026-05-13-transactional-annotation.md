# 기술 학습 기록

## 주제
> 스프링 트랜잭션의 동작 원리와 실무 적용 방식

## 핵심 질문
- `@Transactional` 은 왜 필요할까 ?
- `@Transactional` 은 어떤 방식으로 동작하는 것일까 ? ( 프록시의 이해 )
- 트랜잭션은 언제 시작되고 언제 커밋 또는 롤백되는가 ?
- checked exception 과 runtime exception 에 따라서 롤백 동작은 어떻게 달라지는가 ?
- 스프링 트랜잭션과 데이터베이스 트랜잭션의 상관관계는 ?

## 이미 알고 있는 내용

**`@Transactional` 의 필요성**
- Repository 수준에서 우리는 네이티브 쿼리 / JPA 등을 활용하여 DB 로 쿼리를 전송할 수 있다. 이 때 각각의 쿼리들은 개별 DB 수준 트랜잭션에서 수행되게 된다.
- 만약 여러 쿼리들을 하나의 DB 트랜잭션으로 묶어서 사용하고 싶다면 ? 모든 네이티브 쿼리를 하나로 묶거나, 아니면 직접 DB 트랜잭션을 조작해야 할 필요가 있을 것이다.
- `@Transactional` 은 이를 대체하고자 만들어져서 스프링 수준에서 DB 와 관련된 작업을 하나의 트랜잭션으로 묶어주는 역할을 하게 된다.

**트랜잭션의 동작 원리**
- @Transactional 을 추가하게 되면 스프링 AOP 에 의해 프록시 객체가 IoC 컨테이너에 등록된다.
- 실제로 메서드를 호출하는 시점에 프록시 객체를 활용하게 되므로 트랜잭션과 관련한 기능이 먼저 수행된 뒤 메서드 로직을 수행하게 된다.
- 예외 없이 종료된다면 트랜잭션이 커밋되고, 예외가 발생하게 되면 어떻게 처리되는지 아직 자세히 이해하고 있지 못한 것 같음

## 학습을 통해 정리한 내용

**스프링 AOP**

AOP 란 핵심 관심사를 가로지르는 공통( 횡단 ) 관심사( 트랜잭션, 로그, 보안, 성능 측정 등 )를 모듈화하는 프로그래밍 패러다임이다.

AOP 의 목적은 횡단 관심사를 모듈화하여 핵심 관심사에 영향을 주지 않고도 프로그램에 적용할 수 있도록 하는 것이다.

핵심 개념 4가지
1. Aspect
    - 트랜잭션, 로깅처럼 공통으로 적용될 **기능**
2. Advice
    - **언제 무엇**을 할지
3. Pointcut
    - **어디**에 Aspect 를 적용할지
4. Proxy
    - 프록시가 호출을 가로채는 방식

**트랜잭션의 동작 원리**

트랜잭션을 이해하기 위해선 스프링 AOP 부터 이해할 필요가 있으며 위에서 알아보았다.

애노테이션이 적용되는 순서를 보면 다음과 같게 된다.
0. 스프링이 뜨는 시점에 TransactionAdvisor 가 프록시를 IoC 컨테이너에 등록
1. 클라이언트가 @Transactional 메서드 호출
2. 프록시가 호출을 받음
3. `TransactionAdvisor` 를 통해 트랜잭션이 적용되어야 하는지 확인하고, `TransactionInterceptor` 가 `TransactionAttributeSource` 를 기반으로 `TransactionManager` 를 호출하여 트랜잭션 동작
4. 실제 비즈니스 로직 실행
    - 예외 없이 종료되었다면 커밋 후 호출자에게 반환
    - 예외 발생했다면 롤백 대상 예외인지 판단 후 롤백 또는 커밋 후 예외 재전파

**스프링 트랜잭션과 DB 트랜잭션의 상관관계**

스프링 트랜잭션은 DB 트랜잭션과 별개의 새로운 트랜잭션이 아니다. 스프링은 `TransactionManager` 를 통해서 실제 DB 트랜잭션을 시작하고, 같은 작업 흐름에서 하나의 트랜잭션에
참여할 수 있도록 해주는 **추상화 계층**이다.

결국 트랜잭션 자체는 DB 수준에서 처리된다.

트랜잭션이 처리되는 과정은 다음과 같다.

```text
-> JDBC Connection 획득
-> auto-commit 끄고 트랜잭션 시작
-> SQL 실행
-> commit 또는 rollback
-> Connection 반환
```

즉, 물리적인 DB 트랜잭션은 Connection 에 매달려 있다고 이해할 수 있다.

## 프로젝트 적용

- 단순히 쓰기 작업에 `@Transactional` 이 필요한 것이 아님을 알게 되었다.
- 비즈니스 요구사항을 바탕으로 여러 쓰기 작업이 원자적으로 이루어져야 하는 시점에 @Transactional 이 필요하다.
- 같은 클래스 내부 호출에서는 프록시를 거치지 않기 때문에 AOP 프록시가 적용되지 않게 된다.

## 오늘의 결론

- `@Transactional` 은 스프링 AOP 프록시가 메서드 호출을 감싸며 `TransactionManager` 를 통해 트랜잭션 경계를 명시적으로 관리할 수 있도록 해주는 기능이다.
- JPA 에서는 `EntityManager` 와 persistence context 가 하나의 트랜잭션에 바인딩되어 동작한다.
