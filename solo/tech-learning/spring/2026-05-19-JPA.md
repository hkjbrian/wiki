# 기술 학습 기록

## 주제
> flush 동작, Connection, JPQL / @Query, autoCommit

## 핵심 질문
- flush 가 발생하는 시점
- FlushModeType 은 어디에 저장되고 기본 값은 무엇인가 ?
- flush 는 commit 이 아닌데 어떻게 DB 쿼리 결과를 가져올 수 있는걸까 ?
- Derived Query 에서는 트랜잭션이 어떻게 걸리는가 ? 
- JPQL 은 왜 필요한가 ?
- @Modifying 이란?
- 영속성 컨텍스트를 우회하는 쿼리 실행에서의 주의점
- @Transactional 이 실제로 begin() 을 호출하는가 ?

## 학습을 통해 정리한 내용

**flush 가 발생하는 시점**

flush 가 발생하는 시점을 조금 더 자세히 이해하기 위해서는 FLushModeType 에 대한 이해가 필요하다.

FlushModeType 은 JPA 에서 flush 를 언제 호출할지 정하는 설정값이다. 
JPA 표준으로는 대표적으로 2가지가 있다.
```
FlushModeType.AUTO
FlushModeType.COMMIT
```

두 설정 값 모두 당연하게도 transaction 이 commit 되기 이전에 flush 를 호출하며 차이점은 다음과 같다.
1. AUTO
기본 값이며 쿼리를 실행하기 전 필요하다고 판단되는 경우 flush
2. COMMIT
가능하면 commit 시점까지 flush 를 미뤄둔다.

이를 바탕으로 select 문과 같은 조회문을 실행하기 이전에 flush 가 호출될 수도 있다.

**Connection 의 이해**

Connection 이란 애플리케이션과 DB 사이의 통신 통로로 이해할 수 있다.

JDBC Driver 가 관리하는 객체이며, MySQL 기준으로 서버의 session 과 대응된다.

Connection 객체는 다음과 같은 상태를 갖고 있다.
- autoCommit
- readOnly
- transaction isolation level
- 등등

Connection 에 맞추어 MySQL 측에서는 Server 세션을 가지며 다음과 같은 값들을 갖는다.
- 현재 user
- 현재 database
- autocommit
- transaction isolation
- 등등

그리고 해당 Connection 을 담당하는 Server Thread 1개가 배정되어 해당 Connection 에서 오는 SQL 을 순차적으로 처리하게 된다.

이 Connection 으로 JPA / JDBC 에서의 요청이 전달되면 Thread 가 요청을 처리한다. 
이 때 여러 Connection 의 동시성 문제는 다음 것들로 제어한다.
- transaction id
- row lock
- undo log
- redo log
- MVCC read view

이를 바탕으로 Connection 으로 SQL 이 전달되면 즉시 그 결과를 알 수 있고, 각 Connection 이 각자 개별 DB 를 갖는 것처럼 동작할 수 있게 된다.

**Derived Query 의 변환**
JpaRepository 를 extends 하는 interface 에 메서드를 선언하게 되면 메서드명으로 자동으로 SQL 을 생성할 수 있다.

Derived Query 가 처리되는 순서는 다음과 같다.

Spring 시작 시점에 준비
```
JpaRepository extends 한 인터페이스 발견
-> Spring Data JPA 가 프록시 객체 생성
-> 선언되어 있는 메서드명을 파싱
-> 쿼리 메타데이터 생성
-> 엔티티 타입, 필드명, 조건 구조 등을 검증
-> 쿼리 실행 전략 객체 준비
```

실제 메서드 호출 시점
```
프록시 호출
-> 준비된 쿼리 실행 전략 객체 사용
-> EntityManager 획득
-> CriteriaQuery 또는 JPQL 기반 JPA Query 생성
-> 파라미터 바인딩
-> Hibernate 가 SQL 생성
-> JDBC PreparedStatement 준비
-> 값 바인딩
-> executeQuery()
```

JPA Query 객체를 미리 만들어두지 않는 이유는 Query 객체는 EntityManager 에 종속적이기 때문이다.
EntityManager 는 트랜잭션 / 요청 컨텍스트에 따라서 달라질 수 있기 때문에 JPA Query 를 하나 만들어서 재사용하기 쉽지 않다.

따라서 일반적으로 Derived Query 메서드는 서비스 메서드 수준에서 @Transactional 을 통해 하나의 트랜잭션 내에서 동작하도록 하여 영속성 컨텍스트를 활용할 수 있도록 하는게 일반적이다.

**Modifying 이란?**

@Query 로 JPQL / native Query 를 실행하는 경우 중에서 update / delete 문인 경우 @Modifying 을 붙여주어야 한다.

Spring Data JPA 의 @Query 메서드를 호출하면 대략 다음과 같이 동작한다.
```
프록시가 가로챔
-> 메서드 이름 / @Query / 파라미터 분석
-> EntityManager 로 Query 객체 생성
-> 반환 값을 기준으로 getResultList() 또는 getSingleResult() 를 호출한다.
-> Hibernate 가 JPQL Query 를 SQL 로 변환한다.
-> JDBC PreparedStatement 실행 
```

예를 들어 select는 이런 흐름에 가깝습니다.
```java
@Query("select m from Member m where m.name = :name")
List<Member> findByName(String name);
```

내부적으로는 개념상:

```java
Query query = entityManager.createQuery(
"select m from Member m where m.name = :name");
query.setParameter("name", name);
query.getResultList();
```
반면 update/delete는:
```java
@Modifying
@Query("update Member m set m.name = :name where m.id = :id")
int updateName(Long id, String name);
```
개념상:
```java
Query query = entityManager.createQuery(
"update Member m set m.name = :name where m.id = :id"
);
query.setParameter("name", name);
query.setParameter("id", id);
query.executeUpdate();
```

즉, @Modifying 이 없는 경우 Spring Data JPA 는 @Query 메서드를 조회용으로 해석하려고 하기 때문에, update / delete 문에 대해서 잘못된
실행 경로를 타거나 예외가 발생하게 된다.

**영속성 컨텍스트를 우회하는 경우 주의점**

bulk query / native query 와 같이 Dirty Checking 을 거쳐서 실행되는 것이 아닌 SQL 들은 영속성 컨텍스트와의 불일치를 만들 수 있다.
따라서 이를 일치시켜주는 작업에 유의해야 한다. @Modifying 의 clearAutomatically 를 활용할 수 있고, 직접 em.clear() 를 호출할 수도 있다.

**@Transactional 이 실제로 DB transaction begin() 을 호출하는가 ?**

그렇지 않다.

흐름은 다음과 같다.
```
AOP 프록시가 호출을 가로챈다.
-> TransactionManager 가 트랜잭션 시작 준비
-> EntityMaanger / Connection 을 트랜잭션에 바인딩
-> autoCommit = false 설정
-> 메서드 실행
-> 정상 종료 시 flush 후 commit
-> 예외 발생 시 rollback
-> Connection 상태 복구 이후 반환
```

특히 여기서 조금 더 주의해서 볼 수 있는 부분은 다음과 같다.
- Connection 획득은 첫 DB 접근 시점까지 지연될 수도 있다.
- Dirty Checking 을 통해서 Action Queue 에 변경에 따른 작업( Action 객체 )을 쌓아두다가 flush 시점에 DB 로 전송한다.

따라서 실제로 DB 의 transaction begin 을 호출한다기보다는 Connection 의 autoCommit 설정을 바꾸는 것으로 해당 트랜잭션에서의 작업을 커밋하지 않도록 하는 것으로 작업을 묶는다.

**PreparedStatement 가 사용하는 메서드**
SQL 종류에 따라 다른 메서드로 실행된다.
> executeQuery()

는 조회용, ResultSet 을 반환하는 SQL 에 사용된다.

변경쿼리는 보통
> executeUpdate()

결과로 영향을 받은 row 수를 반환한다.

## 프로젝트 적용

현재 프로젝트에서 Repository / Service 계층을 설계할 때 다음 기준을 적용할 수 있다.

1. 조회 로직은 서비스 계층에 `@Transactional(readOnly = true)` 를 명시한다.
    - 단순 Derived Query 라도 트랜잭션 범위를 명확히 한다.
    - 같은 서비스 로직 안에서 1차 캐시, 지연 로딩, 동일성 보장을 안정적으로 사용할 수 있다.
    - Hibernate flush 최적화 효과도 기대할 수 있다.

2. 변경 로직은 서비스 계층에 일반 `@Transactional` 을 적용한다.
    - 여러 Repository 호출을 하나의 작업 단위로 묶는다.
    - 중간 실패 시 rollback 가능하도록 한다.
    - dirty checking 기반 변경은 `save()` 를 남용하지 않고 엔티티 상태 변경 중심으로 처리한다.

3. `@Modifying` 을 사용하는 update / delete 쿼리는 영속성 컨텍스트 불일치를 주의한다.
    - bulk update / delete 이후 이미 로딩된 엔티티는 오래된 값을 가질 수 있다.
    - 필요한 경우 `clearAutomatically = true` 를 사용하거나, 직접 `EntityManager.clear()` 를 호출한다.
    - bulk query 실행 전에 반영되지 않은 변경이 있다면 `flushAutomatically = true` 도 고려한다.

4. 복잡한 조회는 Derived Query 만 고집하지 않는다.
    - 간단한 조건 조회는 Derived Query 를 사용한다.
    - 메서드명이 길어지거나 join / fetch join / 복잡한 조건이 필요하면 JPQL, QueryDSL, Specification 등을 고려한다.
    - DB 종속 기능이 꼭 필요한 경우에만 native query 를 사용한다.

5. Connection Pool 을 고려하여 트랜잭션 범위를 너무 길게 잡지 않는다.
    - `@Transactional` 이 길어질수록 Connection 을 오래 점유할 수 있다.
    - 외부 API 호출, 긴 연산, 파일 처리 등을 DB 트랜잭션 안에 넣지 않도록 주의한다.

## 오늘의 결론

오늘 학습의 핵심은 JPA 의 변경사항이 실제 DB 에 반영되는 흐름을 계층적으로 이해한 것이다.

JPA 에서 엔티티를 변경하면 즉시 SQL 이 실행되는 것이 아니라, 영속성 컨텍스트가 변경 상태를 관리한다. 이후 flush 시점에 Hibernate 가 dirty checking 을 수행하고, Action Queue 의 작업을 SQL 로 변환하여 JDBC Connection 을 통해 DB 로 전달한다.

flush 는 commit 이 아니다. flush 는 SQL 을 DB 트랜잭션 안으로 보내는 작업이고, commit 은 그 변경을 확정하는 작업이다. 따라서 flush 이후 같은 트랜잭션에서는 변경 결과를 조회할 수 있지만, 다른 트랜잭션에서는 isolation level 과 MVCC 정책에 따라 보이지 않을 수 있다.

`@Transactional` 은 SQL 을 저장하는 버퍼가 아니라 트랜잭션 경계를 관리하는 기능이다. TransactionManager 는 EntityManager / Connection 을 현재 트랜잭션에 연결하고, 정상 종료 시 commit, 예외 발생 시 rollback 을 수행한다.

Connection 은 단순한 통신 통로가 아니라 DB 서버의 session 과 대응되는 실행 단위이다. MySQL 기준으로 각 Connection 은 autocommit, isolation level, 현재 database 등의 session 상태를 가지며, 여러 Connection 은 각자의 transaction 상태에 따라 서로 다른 데이터를 보는 것처럼 동작할 수 있다.

## 면접 질문

**Q. flush 와 commit 의 차이는 무엇인가요?**

flush 는 영속성 컨텍스트의 변경 내용을 SQL 로 만들어 DB 에 전달하는 작업이다. commit 은 현재 DB 트랜잭션의 변경 내용을 확정하는 작업이다. flush 가 발생해도 commit 전이라면 rollback 될 수 있다.

**Q. flush 는 언제 발생하나요?**

대표적으로 트랜잭션 commit 직전, JPQL / Criteria / Derived Query / native query 실행 전, `EntityManager.flush()` 직접 호출 시 발생할 수 있다. 단, flush mode 와 쿼리 종류, 구현체 판단에 따라 실제 동작은 달라질 수 있다.

**Q. select 전에 flush 가 발생할 수 있는 이유는 무엇인가요?**

영속성 컨텍스트에는 변경된 엔티티가 있지만 아직 DB 에 SQL 이 반영되지 않은 상태일 수 있다. 이때 select 를 먼저 실행하면 쿼리 결과가 현재 애플리케이션의 엔티티 상태와 어긋날 수 있으므로, JPA 는 필요하다고 판단하면 select 전에 flush 를 수행한다.

**Q. FlushModeType.AUTO 와 COMMIT 의 차이는 무엇인가요?**

`AUTO` 는 기본값이며 commit 전뿐 아니라 쿼리 실행 전에도 필요하다고 판단되면 flush 할 수 있다. `COMMIT` 은 가능하면 commit 시점까지 flush 를 미룬다. 따라서 `COMMIT` 에서는 같은 트랜잭션 안의 조회 결과가 영속성 컨텍스트의 변경 상태와 다를 수 있다.

**Q. @Modifying 은 왜 필요한가요?**

Spring Data JPA 의 `@Query` 는 기본적으로 조회 쿼리로 해석된다. JPQL update / delete 또는 native 변경 쿼리를 실행하려면 조회 실행 방식인 `getResultList()` 가 아니라 변경 실행 방식인 `executeUpdate()` 를 사용해야 하므로 `@Modifying` 을 붙여야 한다.

**Q. bulk update 가 영속성 컨텍스트를 우회한다는 말은 무슨 뜻인가요?**

bulk update 는 영속성 컨텍스트에 올라온 엔티티 객체의 필드를 변경하는 방식이 아니라 DB 에 직접 update SQL 을 실행하는 방식이다. 따라서 이미 영속성 컨텍스트에 로딩된 엔티티는 DB 변경 이후에도 예전 값을 들고 있을 수 있다.

**Q. Derived Query 는 언제 SQL 로 변환되나요?**

애플리케이션 시작 시점에는 Repository 메서드명이 파싱되고, 쿼리 메타데이터와 실행 전략이 준비된다. 실제 JPA Query 생성, 파라미터 바인딩, Hibernate 의 SQL 생성, PreparedStatement 실행은 Repository 메서드 호출 시점에 이루어진다.

**Q. Repository 메서드에 항상 트랜잭션이 걸리나요?**

`JpaRepository` 기본 제공 메서드에는 `SimpleJpaRepository` 의 트랜잭션 설정이 적용된다. 하지만 직접 선언한 Derived Query 는 기본 트랜잭션이 항상 적용된다고 보기 어렵다. 실무에서는 서비스 계층에 `@Transactional(readOnly = true)` 또는 일반 `@Transactional` 을 명시하는 것이 좋다.

**Q. autoCommit=true 는 트랜잭션이 없다는 뜻인가요?**

아니다. autoCommit=true 는 SQL statement 하나가 실행될 때마다 자동으로 commit 된다는 뜻이다. 즉 트랜잭션이 없는 것이 아니라, statement 단위의 짧은 트랜잭션이 자동으로 시작되고 종료되는 것으로 이해할 수 있다.

**Q. Connection 은 어떤 역할을 하나요?**

Connection 은 애플리케이션과 DB 사이의 통신 통로이며, MySQL 기준으로 서버의 session 과 대응된다. Connection 은 autoCommit, readOnly, isolation level 같은 상태를 가지고, DB 서버도 session 별로 현재 database, autocommit, transaction 상태 등을 관리한다.

**Q. PreparedStatement 의 executeQuery 와 executeUpdate 는 어떻게 다른가요?**

`executeQuery()` 는 select 처럼 `ResultSet` 을 반환하는 조회 쿼리에 사용한다. `executeUpdate()` 는 insert, update, delete 같은 변경 쿼리에 사용하며, 결과로 영향을 받은 row 수를 반환한다.
